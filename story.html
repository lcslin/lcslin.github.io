機器自省論 兩器相詢，一稿往返

兩器相詢，往返一稿，程式遂精——此非人力所為。一場令 Claude 與 Codex 互相問答之實驗，正悄然改變程式碼誕生之道。

Jason CS Lin 撰 丙午年四月三十日 約六分鐘閱畢

丙午之秋，有人授命於機，令謄一程式，藏諸磁碟。此後無人復命，兩器自相問答，頃刻間交還一份較原稿潔凈數倍之作。全程不逾片刻，人未嘗涉其間。

緣起

兩器並立，各司其職

Anthropic 之 Claude，與 OpenAI 之 Codex，各擅其職，久行於工程師之案牘間。前者善解語意，能生程式；後者嚴於稽查，以精確著稱。往昔二者各行其是——工程師先令 Claude 作程式，再持其稿就 Codex 審之，閱所評而決取捨。中間一環，仍賴人手。

今有人欲去此一環。

機制

封鎖一紙，循環自止

此機制之核心，出人意外之簡：一微小之文字檔而已。

每逢 Claude 寫畢，有一鉤子（hook）暗自發動，將最新程式送入 Codex 審閱。審畢，以中文呈現意見，末附一令：「請依所評修正，僅限一次。」

系統同時於磁碟留一封鎖檔，記「此程式正在修正循環中」。待 Claude 修畢、再度寫入，鉤子復起，見封鎖檔在，乃行最終審查，繼而悄然刪之——宣告循環告終，不復催修。

試驗

質數之程，故意為劣

為驗此流水線，試者故意寫就一段多有瑕疵之質數程式：未用之引入、效能不彰之迴圈、冗餘之比較式、裸露之全域副作用——皆程式習見之積病。

原稿 — 積病諸多

import math

data = []

def check(n):
    if n == 1:
        return False
    for i in range(2, n):
        if n % i == 0:
            return False
    return True

for x in range(0, 100):
    r = check(x)
    if r == True:
        data.append(x)

print(data)

改後 — Codex 審畢

import math


def is_prime(n: int) -> bool:
    if n < 2:
        return False
    for i in range(2, math.isqrt(n) + 1):
        if n % i == 0:
            return False
    return True


if __name__ == "__main__":
    primes = [x for x in range(100)
              if is_prime(x)]
    print(primes)

Codex 審評，一針見血。演算法效能自 O(n) 化為 O(√n)；函式命名由模糊之 check 易為 is_prime；多餘引入得以正用；全域執行碼納入保護之中。

觀察

不為之道，重於所為

此流水線最可觀者，或非其所為，而在其刻意不為之處。

系統設計，最多僅行一輪自動修正。縱第二次 Codex 審查仍有所見，亦不自行繼續，而是將控制之權還諸工程師。此硬性終止之設計，乃對機器能力邊界之清醒自知：明確之事委諸機器，需判斷之事歸於人類。

技術上，此整合之實作成本極低：不足五十行之 Shell Script、一份 JSON 設定、一個暫存之文字檔。無獨立伺服器，無複雜串接，全行於工程師之本機。此零基礎設施之特性，令其幾乎可由任何已用 Claude Code 之工程師，以分鐘為單位部署之。

展望

此後之問

生成與審查既皆可自動，工程師之職責正悄然位移。日常之程式品質守門，既可外包於機器；人之心力，遂得釋放，以謀更難自動化之事——架構之決斷、用者之需求，以及究竟欲造何物。

此實驗規模雖小，所問卻大：當諸機器開始彼此問答，而非獨與人類周旋，吾人對「軟體開發」之想像，是否亦需一番自省？

整個自動修正循環僅含兩輪——初審、一次修正、終審。逾二輪者，刻意棄之，以免系統陷入無謂之自我修正。

自動修正四步

一　謄寫

Claude 生程式，藏諸磁碟

二　初審

鉤子發動，Codex 評其正確、安全、可讀，建封鎖檔

三　修正

Claude 閱評，自行修正，僅此一次

四　終審

鉤子再起，行最終審查，刪封鎖檔，循環告終

令兩器相互問答之意，非在打造永不停歇之自動機器，而在建立一道篩網，先摘低垂之果，再讓人手介入。
